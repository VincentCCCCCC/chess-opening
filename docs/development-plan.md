# 国际象棋开局记录与练习小游戏开发阶段方案

本文档基于 `docs/design-guideline.md` 制定，用于指导 V1 个人可用版的分阶段开发。每个阶段都必须通过本阶段验收后，才能进入下一阶段；如果验收发现问题，优先在当前阶段修正，不把未完成事项滚入后续阶段。

## 1. 阶段推进原则

- 先完成可运行骨架，再开发业务闭环，最后补齐体验、统计、授权和测试。
- 每个阶段都产出可检查的代码、游戏场景或文档，不只产出概念设计。
- 任何实现不得引入 guideline 明确列为第一版非目标的功能，例如云同步、PGN 导入导出、AI 分析、在线对弈、复杂排行榜。
- 棋盘、棋规、本地数据、练习状态机是核心风险，应尽早实现并单独验证。
- 进入下一阶段前必须完成本阶段的“完成标准”和“交接检查”。

## 2. 全局完成标准

V1 完成时，应满足以下条件：

- 用户可以创建、编辑、删除开局，并为开局设置练习方和标签。
- 用户可以在棋盘上录入、编辑、删除线性变式，非法走法不会写入主数据。
- 用户可以浏览变式，并跳转到任意一步局面。
- 用户可以进行单条、顺序、随机练习；随机只打乱变式顺序，不打乱变式内部走法。
- 练习时只由用户走开局设置的练习方，对手方由程序按记录自动走。
- 用户走错后可以重试或查看正确走法；查看后仍要求重新走一遍。
- 每条变式能记录全对、完成但有错、放弃等练习结果。
- 本地存储带 `schemaVersion`，能初始化默认数据，并对损坏变式做阻断提示。
- cburnett 棋子素材来源和许可证说明已保留。
- DevTools 或真机完成核心路径测试。

## 3. Phase 0：需求冻结与工程决策

### 目标

把 guideline 转换成明确的 V1 开发范围，避免后续边做边扩。

### 要做的事

- 确认 V1 只做个人本地使用，不做账号、云同步、分享、PGN 导入导出。
- 确认场景范围：开局库、开局详情、变式编辑、变式浏览、练习、设置；设置场景可只保留占位或后置。
- 确认工程形态：仓库根目录保留 `docs/`，小游戏代码放入 `miniprogram/`，根目录放 `project.config.json`。
- 确认 Phase 1 先用 `game.js` 搭建稳定小游戏入口；后续业务模块如需使用 TypeScript，必须先验证 DevTools 对小游戏 TypeScript 的编译链。
- 确认棋规服务优先使用成熟库，避免自行实现完整国际象棋规则。
- 建立 V1 不做清单，开发中发现新想法先记录，不直接插入当前版本。

### 产出物

- 本开发阶段方案。
- 明确的 V1 范围和不做范围。
- 工程目录决策。

### 完成标准

- V1 场景、渲染模块、服务、数据模型范围与 guideline 一致。
- 没有把 V1.1、V1.2、V2、V3 功能提前纳入必做范围。
- 已确认下一阶段可以开始创建小游戏基础工程。

### 交接检查

- 如果仍无法确定 `compileType: game`、`miniprogramRoot`、入口文件和场景范围，则不能进入 Phase 1。

## 4. Phase 1：小游戏基础工程与可运行骨架

### 目标

搭建一个能被微信 DevTools 以“小游戏”模式打开的最小 Canvas 工程。

### 要做的事

- 创建根目录 `project.config.json`，设置 `compileType` 为 `game`，并设置 `miniprogramRoot` 指向 `miniprogram/`。
- 创建 `miniprogram/game.js` 和 `miniprogram/game.json`，不要创建或依赖 `app.json`、页面路由、WXML、WXSS。
- 在 `game.js` 中创建 Canvas、适配设备尺寸和像素比，并渲染一个可见的开局库占位场景，避免黑屏。
- 在 Canvas 内预留 V1 场景入口：开局库、开局详情、变式编辑、变式浏览、练习、设置。
- 创建基础目录：`scenes/`、`ui/`、`services/`、`models/`、`styles/`、`assets/`。
- 设置基础主题 token，先采用 guideline 的安静工具风格色彩。
- 实现最小触摸处理，至少能在占位场景间切换或返回开局库，确保主循环和输入正常。

### 产出物

- 可被 DevTools 以小游戏模式导入的基础工程。
- Canvas 首屏可见，不再请求 `app.json` 或页面文件。
- 初始主题 token 与占位场景渲染。

### 完成标准

- DevTools 可以打开项目，且没有 `game.json` 缺失或页面路径缺失错误。
- `miniprogram/game.json` 和 `miniprogram/game.js` 存在并能启动。
- 首屏 Canvas 能显示“开局库”骨架内容，触摸入口有响应。
- 根目录和小游戏代码目录职责清晰，`docs/` 不被放进小游戏代码根。

### 交接检查

- 如果 `compileType`、`game.json`、Canvas 首屏或触摸输入存在问题，必须在本阶段修正。

## 5. Phase 2：基础模型、本地存储与数据迁移

### 目标

先把数据结构和本地存储稳定下来，为后续游戏场景和练习逻辑提供可靠基础。

### 要做的事

- 在 `models/` 定义核心类型：`AppData`、`Opening`、`Variation`、`RecordedMove`、`VariationStats`、`PracticeSession`、`PracticeVariationResult`、`PracticeMistake`。
- 定义标准初始 FEN 常量。
- 在 `services/storage/` 封装本地存储访问，使用 key `chessOpening.appData.v1`。
- 实现默认数据初始化：无数据时创建 `schemaVersion`、空开局、空变式、空练习记录、默认设置。
- 实现最小数据校验：检查 schemaVersion、数组字段、Opening/Variation 的必要字段。
- 实现写入前校验和整包保存。
- 预留迁移入口：旧版本数据先返回明确错误或迁移结果，不静默覆盖。
- 定义删除策略：V1 可真删除，但删除开局时同步删除其变式和关联练习记录。

### 产出物

- 数据模型类型定义。
- 本地存储服务。
- 默认数据初始化和保存能力。
- 数据损坏时的可识别错误类型。

### 完成标准

- 可以在游戏场景或调试入口中创建、读取、更新、删除一条 Opening 测试数据。
- 刷新或重新打开小游戏后数据仍存在。
- 空数据时不会报错，能返回默认结构。
- 损坏数据不会被直接覆盖，能返回明确错误供 UI 提示。

### 交接检查

- 如果没有稳定存储服务，不进入开局库场景开发，避免场景直接调用 `wx.setStorageSync` 形成散乱数据写入。

## 6. Phase 3：棋规服务与走法记录能力

### 目标

封装棋规能力，让场景和渲染模块不直接处理复杂国际象棋规则。

### 要做的事

- 在 `services/chess/` 封装棋规服务。
- 支持从 FEN 创建局面、生成合法走法、尝试走法、输出新 FEN、输出 SAN。
- 支持普通走法、吃子、王车易位、吃过路兵、升变、将军判断、不能走后自将。
- 实现 `TryMoveInput` 和 `TryMoveResult`，非法走法返回 `illegal_move`，缺少升变选择返回 `promotion_required`，回合错误返回 `wrong_turn`。
- 生成 `RecordedMove`，保存 `ply`、`side`、`from`、`to`、`promotion`、`san`、`fenAfter`。
- 增加从初始局面重放变式的能力，用于浏览、练习初始化、数据异常检测。
- 记录第三方棋规库许可证信息，如果引入库。

### 产出物

- 棋规服务接口和实现。
- 走法记录生成逻辑。
- 变式重放校验逻辑。

### 完成标准

- 能正确处理基础开局走法，例如 `e2-e4`、`e7-e5`、`g1-f3`。
- 能正确拒绝非法走法和走后自将的走法。
- 能识别升变需要选择，并在选择后生成正确记录。
- 能重放一条合法变式并得到最后局面 FEN。
- 能识别一条损坏或不合法变式，供列表标记“数据异常”。

### 交接检查

- 如果棋规服务还不可靠，不进入棋盘录入阶段，因为变式编辑依赖它保证主数据正确。

## 7. Phase 4：棋盘渲染模块与基础视觉系统

### 目标

实现所有模式共用的棋盘交互和展示基础。

### 要做的事

- 创建 `ui/chess-board/` 或等价渲染模块，封装棋盘绘制、命中检测和高亮逻辑。
- 实现棋盘 8x8 渲染，支持白方在底部和黑方在底部。
- 支持基于 FEN 渲染棋子。
- 引入 cburnett 棋子素材到 `miniprogram/assets/pieces/cburnett/`。
- 实现点击选中棋子、显示合法目标格、点击目标格触发 `movetry`。
- 支持 `selected`、`legal-target`、`last-move-from`、`last-move-to`、`expected-from`、`expected-to`、`wrong-from`、`wrong-to`、`check` 高亮。
- 按 guideline 的高亮优先级处理冲突。
- 创建 `promotion-picker` 渲染模块，支持后、车、象、马选择，默认高亮后。
- 创建基础 `move-list` 渲染模块，用于后续编辑和浏览复用。

### 产出物

- 可复用棋盘渲染模块。
- 升变选择渲染模块。
- 基础走法列表渲染模块。
- 棋子素材目录。

### 完成标准

- 棋盘显示清晰，默认尺寸在手机视口内尽量大。
- 棋盘方向能随 `orientation` 切换。
- 点击操作稳定，不依赖拖拽。
- 合法目标、最后一步、错误提示、正确答案提示能正确显示。
- 升变场景不会在用户选择前提交走法。

### 交接检查

- 如果棋盘点击、方向或高亮不稳定，不进入变式编辑阶段。

## 8. Phase 5：开局库与开局详情闭环

### 目标

完成开局维度的管理，并建立进入变式和练习的核心入口。

### 要做的事

- 实现开局库场景：展示所有开局、空状态、新建开局入口。
- 开局卡片展示名称、练习方、标签、变式数量、最近练习时间、最近状态。
- 实现新建和编辑开局：名称、练习方、标签、颜色可选。
- 实现删除开局，删除前二次确认，并同步处理其变式和练习记录。
- 实现开局详情场景：顶部开局信息，中部棋盘预览，底部变式列表和固定操作区。
- 开局详情固定操作区提供“新建变式”“顺序练习”“随机练习”。
- 开局无变式时显示 guideline 指定空状态文案和“新建变式”按钮。
- 变式列表先支持展示已有变式摘要；编辑和浏览跳转可在后续阶段补全。

### 产出物

- 开局库场景。
- 开局详情场景。
- Opening 增删改查闭环。
- 空状态和删除确认。

### 完成标准

- 用户可以从空数据创建第一个开局。
- 创建后可以进入开局详情。
- 修改开局名称、练习方、标签后能保存并持久化。
- 删除开局需要二次确认，确认后相关数据不残留在列表中。
- 开局详情能根据练习方显示棋盘方向。

### 交接检查

- 如果 Opening 数据管理不稳定，不进入 Variation 编辑，否则后续关联关系会混乱。

## 9. Phase 6：变式编辑闭环

### 目标

让用户可以在棋盘上录入和保存一条线性变式。

### 要做的事

- 实现变式编辑场景，棋盘作为画面中心。
- 支持从标准初始局面开始录入双方走法。
- 每一步通过棋规服务校验，合法才写入临时走法序列。
- 非法走法不写入主数据，只显示错误提示。
- 支持升变选择、撤销、重做、清空、保存。
- 支持显示当前轮到哪方走和最近一步走法。
- 支持保存时输入变式名称和备注；没有名称时自动生成“变式 N”。
- 允许保存一手或几手的短变式。
- 支持编辑已有变式：加载原走法，修改后重新保存。
- 删除变式需要二次确认。

### 产出物

- 变式编辑场景。
- Variation 创建、编辑、删除能力。
- 录入走法临时状态管理。

### 完成标准

- 用户可以从开局详情创建一条至少一手的变式。
- 保存后的变式出现在开局详情列表中，并展示走法摘要和总半回合数。
- 非法走法、错误回合、未选择升变都不会写入保存数据。
- 撤销、重做、清空不会造成棋盘局面和走法列表不一致。
- 编辑已有变式后，`updatedAt` 和统计关联保持合理。

### 交接检查

- 如果变式录入无法保证合法性，不进入浏览和练习阶段。

## 10. Phase 7：变式浏览与回放

### 目标

提供只读复盘能力，明确区分浏览模式和编辑模式。

### 要做的事

- 实现变式浏览场景，加载指定变式并从初始局面开始展示。
- 支持上一步、下一步、回到开头、跳到结尾。
- 支持点击走法列表中的任一步，棋盘跳转到对应局面。
- 浏览场景不直接修改变式主数据。
- 提供“练习此变式”和“编辑变式”入口。
- 对无法重放的变式显示“数据异常”，提供删除或编辑入口，不进入练习。

### 产出物

- 变式浏览场景。
- 变式回放控制逻辑。
- 数据异常展示和阻断逻辑。

### 完成标准

- 用户可以从开局详情进入任意变式浏览。
- 任意跳步后棋盘局面、最后一步高亮、走法列表选中态一致。
- 浏览场景不会写入变式数据。
- 数据异常变式不会进入练习。

### 交接检查

- 如果走法重放与棋盘展示不一致，不进入练习阶段。

## 11. Phase 8：单条练习与练习状态机

### 目标

先完成最小练习闭环，再扩展到顺序和随机练习。

### 要做的事

- 实现练习场景基础结构：顶部开局名、当前变式、进度，中部棋盘，底部提示与操作。
- 实现 `PracticeState` 状态机：`loading`、`auto_playing_opponent`、`waiting_user_move`、`showing_correct_feedback`、`showing_error_feedback`、`showing_answer`、`variation_completed`、`session_completed`、`abandoned`。
- 单条练习进入后，根据开局练习方决定用户走白方或黑方。
- 黑方练习时，如果变式从白方第一手开始，程序先自动执行白方第一手。
- 用户走法按 `from`、`to`、`promotion` 与记录走法完全比对。
- 走对时执行用户走法，显示“正确”，再延迟 300-600ms 自动执行对手走法。
- 走错时不推进进度，不写入主线，回到走错前局面，并显示“再试一次”“查看正确走法”。
- 查看正确走法时高亮正确起点和终点，用户确认后仍需重新走一遍。
- 完成单条后记录 `perfect` 或 `completed_with_errors`，包含错误步信息和用时。
- 支持放弃练习并记录 `abandoned`。

### 产出物

- 单条练习场景闭环。
- 练习状态机。
- 单条练习结果写入。

### 完成标准

- 白方练习和黑方练习都能正确开始。
- 程序自动走对手方时禁止用户操作，避免状态错乱。
- 走错后可以重试，也可以查看正确走法。
- 查看答案不会直接把错误当作正确完成。
- 完成后能更新变式统计：练习次数、全对次数、连续全对、最近练习时间、最近错误。

### 交接检查

- 如果单条练习状态机不稳定，不进入顺序和随机练习阶段。

## 12. Phase 9：顺序练习、随机练习与练习结果场景

### 目标

把单条练习扩展为开局内多变式练习轮次。

### 要做的事

- 实现顺序练习：按当前变式列表顺序，默认创建时间升序。
- 实现随机练习：进入练习时生成固定随机顺序，本轮中不再变化。
- 每条变式内部仍保持原始走法顺序。
- 一条变式完成后展示轻量完成态，用户点击“下一条”进入下一条。
- 全部完成后展示整轮结果：总变式数、全对条数、有错条数、总用时。
- 提供“再练一轮”和“返回开局”。
- 更新 Opening 的最近练习时间和列表状态。
- 处理空变式：禁止开始练习并提示先新建变式。

### 产出物

- 顺序练习入口和流程。
- 随机练习入口和流程。
- 练习结果渲染模块或场景片段。
- PracticeSession 写入。

### 完成标准

- 顺序练习按列表顺序完成全部变式。
- 随机练习在一轮内顺序固定，刷新前后如需恢复则有明确策略；V1 可选择不恢复中断轮次。
- 结果统计与每条变式结果一致。
- 变式完成后不会自动跳过用户确认，避免用户看不到本条结果。

### 交接检查

- 如果多变式练习结果无法准确写入，不进入最终打磨阶段。

## 13. Phase 10：体验打磨、授权与边界情况

### 目标

补齐 V1 必需的边界、文案、授权和视觉一致性。

### 要做的事

- 按 guideline 统一按钮层级和文案：主要按钮、次要按钮、危险按钮。
- 统一空状态文案：无开局、无变式。
- 棋盘周围减少大段说明，保持棋盘优先。
- 检查所有危险操作二次确认：删除开局、删除变式、清空当前录入。
- 对短变式、对手无下一步、黑方起始自动步、数据损坏做专项处理。
- 补充 `LICENSES/cburnett.md`，保留 cburnett 来源和 GPLv2+ 说明。
- 如果引入棋规库，也补充对应许可证记录。
- 检查首屏、详情、编辑、浏览、练习场景在常见手机宽度下可用。
- 收敛所有 V1 不做功能入口，避免出现不可用按钮。

### 产出物

- 统一后的 UI 文案和按钮层级。
- 授权说明文件。
- 边界情况处理。
- 移动端布局修正。

### 完成标准

- 所有 guideline 规定的错误和边界情况都有明确处理。
- 游戏场景没有无法完成的入口或误导性功能。
- 棋子素材和第三方库授权可追踪。
- 核心场景在 DevTools 模拟器中视觉稳定。

### 交接检查

- 如果授权或数据损坏处理缺失，不进入发布前测试。

## 14. Phase 11：集成测试、回归与 V1 验收

### 目标

验证核心路径完整可用，确认 V1 达到个人可用标准。

### 要做的事

- 执行空数据路径测试：首次打开 -> 创建开局 -> 创建变式 -> 浏览 -> 练习。
- 执行白方练习路径测试。
- 执行黑方练习路径测试，特别检查进入后自动执行白方第一步。
- 执行非法走法测试：非法移动、错误方走棋、升变未选择、走错练习步。
- 执行编辑路径测试：编辑开局、编辑变式、删除变式、删除开局。
- 执行顺序练习和随机练习测试。
- 执行数据持久化测试：关闭并重新打开后数据仍正确。
- 执行数据异常测试：损坏变式不进入练习，能标记异常。
- 在 DevTools 和至少一个真机环境中完成核心路径验证。
- 根据 `docs/design-guideline.md` 第 19 节逐条做设计验收。

### 产出物

- V1 验收记录或测试清单。
- 已修复的回归问题。
- 可继续迭代到 V1.1 的代码基线。

### 完成标准

- 核心路径无阻断问题。
- 第 19 节设计验收标准全部通过，或有明确记录说明为何延后。
- 所有 V1 必须完成项都已实现。
- 可以基于当前版本开始个人真实录入和练习。

### 交接检查

- 只有 Phase 11 通过后，才进入 V1.1 效率增强；否则继续修复 V1。

## 15. 推荐开发顺序总览

```txt
Phase 0  需求冻结与工程决策
Phase 1  小游戏基础工程与可运行骨架
Phase 2  基础模型、本地存储与数据迁移
Phase 3  棋规服务与走法记录能力
Phase 4  棋盘渲染模块与基础视觉系统
Phase 5  开局库与开局详情闭环
Phase 6  变式编辑闭环
Phase 7  变式浏览与回放
Phase 8  单条练习与练习状态机
Phase 9  顺序练习、随机练习与练习结果场景
Phase 10 体验打磨、授权与边界情况
Phase 11 集成测试、回归与 V1 验收
```

## 16. 每阶段通用检查清单

进入下一阶段前，统一检查：

- 是否符合 guideline 的产品定位和 V1 范围。
- 是否仍围绕棋盘完成核心操作。
- 是否遵守开局级练习方设定。
- 是否明确区分录入、浏览、练习三个模式。
- 是否没有把错误走法写入变式主数据。
- 是否没有引入第一版不做的复杂功能。
- 是否有空数据、错误数据、删除确认等基本边界处理。
- 是否能在 DevTools 中运行并手动验证本阶段核心路径。
